Your code currently treats RMC as the authority for whether you have a valid fix: const fixStatus = parts[2]; // A = valid, V = void if (fixStatus !== "A") { setStatus("RMC received but fix is not valid", "warn"); return; } If your AVL kept saying: "RMC received but fix is not valid" while your GPS program showed a perfectly good fix, there are a few likely explanations. Possibility 1 (my top suspicion): The dongle was outputting V in RMC but valid GGA sentences Some inexpensive USB GPS receivers do something like this: $GPRMC,...,V,... but simultaneously output $GPGGA,...,1,... where: RMC = V (Void) GGA Fix Quality = 1 (Valid GPS) Technically that's inconsistent, but it happens. Your parser accepts the GGA fix... fixQuality = 1 ...then a fraction of a second later an RMC sentence comes in with V and overwrites the status with: RMC received but fix is not valid even though the coordinates themselves may still be perfectly usable. Possibility 2 The receiver had not yet received enough satellites to compute speed/course. Some GPS chips won't mark RMC as valid until they know: heading speed even though latitude/longitude are already valid. Possibility 3 The receiver was indoors. Many cheap GPS dongles will alternate like this: GGA valid RMC void GGA valid RMC void GGA valid RMC valid for several minutes. I actually want to make your parser smarter. Instead of saying: "RMC invalid" I'd rather have the AVL behave like this: RMC invalid ↓ Do we already have a recent GGA fix? YES ↓ Ignore the bad RMC. Keep using the last known fix. So instead of immediately returning: if (fixStatus !== "A") { setStatus("RMC received but fix is not valid"); return; } I'd change it to something like: if (fixStatus !== "A") { if (lastFix && (Date.now() - lastFix.gpsTime) < 5000) { // We already have a fresh GGA fix. // Ignore this bad RMC. return; } setStatus("Waiting for valid GPS fix...", "warn"); return; } That way one flaky RMC sentence doesn't make the AVL think GPS has been lost. Another improvement I'd like to make I'd also stop keying status messages off a single sentence type. Instead of showing: RMC received but fix is not valid I'd rather show something dispatch-friendly: GPS Acquiring Satellites... or External GPS Connected Waiting for Position Fix... Those messages describe what's actually happening and avoid making it look like the GPS has failed when it's simply still acquiring or sending inconsistent NMEA sentences.
